# Reasoning: Portfolio Update

This document captures the thinking behind the prompt. It exists so a reader can understand not just *what* the prompt is, but *why* it ended up this way.

## Goal

Immediately document and publish the completed project to the developer's portfolio — same session as deployment, no delay between shipping and recording.

## Iteration history

Single prompt, sent immediately after the GitHub Pages URL was confirmed live. The structured key-value format (Name / Description / Live URL / Tech Stack / Features) mirrors a professional project brief, making it easy for the skill to parse without ambiguity.

## Failure modes the final version handles

By including both the live URL and the GitHub URL, the prompt ensures the portfolio entry has two distinct links — the demo and the source — which is standard for a developer portfolio. Leaving either out would have produced an incomplete entry.

## Outcome

Grant Matcher Pro added to the portfolio with full project details, live link, and tech stack listed. Followed up with a request to include the live URL in the README as well.

## What I'd change next

Could add a one-line "what I learned" field to the prompt so the portfolio entry captures the learning outcome automatically, not just the project facts.
